01 Suite Overview

The ProjectLens suite is a development-portfolio system for showcasing funded projects (NGO, government, or grant-based work) on a WordPress site. It consists of two plugins that work together:

PluginSlug / Text DomainVersionRole
ProjectLens projectlens 2.1.0 Core plugin. Registers the pl_project custom post type, the segment-based content model, full-page frontend templates, styling controls, role-based access control, global settings, and demo data.
ProjectLens Project Editor pl-project-editor 1.2 Add-on. Adds a [pl_project_editor] shortcode that renders a full project-editing interface on any frontend page, for users who hold the pl_edit_projects capability but do not have access to wp-admin.

Dependency: ProjectLens Project Editor requires the core ProjectLens plugin to be active. If ProjectLens is missing, the editor plugin displays an admin notice and does not load.

What ProjectLens Projects Look Like

Each project is a single post of type pl_project, displayed on its own full-page template (the active theme's header/footer are bypassed). The page uses a 70/30 split layout:

  • Left panel (70%) — a vertical stack of "segments" (descriptive text, media sliders, metadata tables, team cards, timelines, document lists), each shown in summarised form.
  • Right panel (30%) — a detail panel that displays the full content of whichever segment the visitor clicks "Show Detail" on. On mobile, this becomes a slide-over view with a back button.

All projects are listed on an archive page with category/status filter pills and pagination.

04 Project Segments

A project's body content is built from an ordered list of segments, stored in post meta key _pl_segments as an array. Each segment has a type, a title, an enabled flag, and type-specific fields.

Type keyLabelFields storedSummary viewDetail view
descriptiveDescriptive subtitle, content (rich text), detail_content (rich text) Subtitle + content truncated to project word limit, with a "Show Detail" button Full detail_content (falls back to content if empty)
mediaMedia Slider slides[] — each with media_id, media_type, video_url, caption, description, geo_lat, geo_lng, geo_label Horizontal thumbnail strip; videos show a play icon; YouTube/Vimeo URLs auto-fetch a thumbnail Full-size viewer per slide, including embedded video and a Leaflet map for any slide with geo-coordinates
metadataProject Metadata fields[] — each a label/value pair First 5 fields as a definition list; "Show all N fields" button if more than 5 All fields as a full-width definition list
teamTeam & Stakeholders members[] — each with avatar_id, name, role, email, bio Row of clickable avatars with name/role Card per member with larger avatar, email link, and bio

Segment Admin Controls

  • Drag handle — reorder segments via jQuery UI Sortable.
  • Title field — optional heading shown above the segment on the frontend.
  • Visible checkbox — corresponds to enabled; hidden segments are skipped entirely on the frontend.
  • Collapse toggle — purely a UI convenience in the admin editor (stored as collapsed).
  • Delete — removes the segment and all its sub-items.

YouTube / Vimeo Thumbnails

For media segments where a slide's media_type is video_url and no media_id is set, ProjectLens automatically derives a thumbnail:

  • YouTube — thumbnail URL is built directly from the video ID (https://img.youtube.com/vi/<id>/hqdefault.jpg), no API call needed.
  • Vimeo — the plugin calls Vimeo's oEmbed endpoint and caches the result in a transient for 24 hours (pl_vimeo_thumb_<id>).

07 Frontend Templates (Single & Archive)

Both the single project page and the projects archive use dedicated full-page templates that bypass the active theme (via the template_include filter). Each template calls wp_head() and wp_footer() so plugins/analytics still work, but the theme's header.php/footer.php are not used.

Single Project Template (single-pl_project.php)

  • Top bar — "← All Projects" link back to the archive, an optional sticky nav menu (Section 10), and the site name.
  • Header block (if show_header is enabled) — featured image as a background, status badge, location, funder, date range, title, and short description.
  • Main 70/30 layout: #pl-summary-panel (70%) renders every enabled segment; #pl-detail-panel (30%) initially shows a "Project Overview" block built from Project Settings fields.
  • Embedded JSON — all segment data is embedded as <script id="pl-segments-data" type="application/json">. projectlens.js reads these to power the detail panel, sliders, and Leaflet maps without further AJAX calls.

Archive Template (archive-pl_project.php)

  • Top bar — optional sticky nav menu (archive location) + site name.
  • Archive header — title with a live count of matching projects.
  • Filter bar (if enabled in Global Settings): Category pills (pl_category terms) and Status pills (pl_status terms).
  • Project grid — one card per project: thumbnail with status badge overlay, category tags, title, short description, location/funder/date meta row, and a "View Project →" link.
  • Pagination — numbered page links based on per_page from Global Settings.

Query Loop / FSE Compatibility

Because show_in_rest is enabled, the Query Loop block can list pl_project posts on full-site-editing pages. ProjectLens adds a pre_get_posts hook that ensures a secondary Query Loop query requesting pl_project is honoured even if WordPress would otherwise override it with the main query.

11 Demo Data

From Projects → Global Settings, administrators can install a single, fully-populated sample project that exercises every segment type and the full 70/30 layout — useful for previewing styling changes or training new editors.

  • Install Demo Project — creates one demo pl_project post with sample segments, settings, and styling. Its post ID is stored in an option so the plugin can find it later.
  • Remove Demo Project — permanently deletes the demo project and all its data. Requires confirmation.
  • If a demo project already exists, the "Install" action is blocked with a warning notice.
  • Installing the demo links directly to "View it →" on success.

Tip: the demo project is a normal pl_project post — it can be edited, locked, or used as a starting template by duplicating its content with a page-duplication plugin.

12 Editor Plugin Overview

ProjectLens Project Editor (slug pl-project-editor, version 1.2) adds a single shortcode, [pl_project_editor], that renders a complete project-editing interface on any frontend page. It is designed for users — typically field staff, content coordinators, or NGO partners — who need to update project content but should not have access to wp-admin.

Capabilities of the Editor

  • Lists every pl_project post the current user is allowed to edit.
  • Renders a tabbed editor mirroring the wp-admin meta boxes.
  • Saves changes via AJAX (no page reload).
  • Provides a scoped media library: any eligible user can browse and use any attachment, but can only delete attachments they personally uploaded.

Explicit Limitations

  • Cannot create new projects. The project dropdown only lists existing posts.
  • Cannot delete projects. No delete action exists anywhere in the editor UI.
  • Does not expose Styling controls. Colors, fonts, border radius, etc. remain admin-only.
  • Non-administrators cannot add or remove segments, change the project title, edit Project Abstracts, or toggle document download/viewer flags — see Section 14.

Compatibility Notes

  • REST media upload fix — the plugin filters user_has_cap to grant upload_files at the REST layer specifically to users who already hold pl_edit_projects.
  • WP-Optimize minifier fix — the plugin excludes its own script handle (ple-editor) from WP-Optimize's script concatenation via the wpo_minify_exclude_handles filter.

Editor Shell Structure

  • Project selector — a dropdown of all publish/draft/private projects, ordered by title. Locked projects are hidden from non-administrators entirely (shown to admins with a 🔒 icon).
  • Preview button — opens the selected project's live page in a new tab.
  • Project Title field — editable only by administrators; read-only for everyone else.
  • Tabs: Project Segments (always visible); Project Abstracts (administrators only); Project Settings (always visible, contains the Feature Image control).
  • Save Changes button — triggers the AJAX save (Section 15).

Assets are enqueued only on pages where the shortcode is present and the visitor is logged in with pl_edit_projects — keeping the footprint minimal elsewhere on the site.

The frontend editor enforces a stricter, two-tier permission model on top of the base pl_edit_projects capability check:

Segment-count guard: because non-administrators cannot add/remove segments and the server rejects any save where the segment array's length changed, reordering segments as a non-admin will save individual segment edits but the order itself is only reliably persisted when performed by an administrator.

16 Media Library Behaviour

Browsing

Every user with pl_edit_projects sees the entire site's media library when picking images, videos, or documents — uploads are not filtered by author for browsing/selection purposes. This is intentional: it lets collaborators reuse images other team members have already uploaded for the same project.

Uploading

New uploads go through the standard WordPress media uploader (wp.media) and the REST media endpoint. The user_has_cap filter ensures users with only pl_edit_projects aren't blocked by the default upload_files requirement.

Deleting

Deletion is strictly self-service: the delete button/action is only effective for files the current user uploaded themselves (post_author === current_user_id), checked server-side on every request regardless of role. Deletion is permanent (wp_delete_attachment(..., true) — bypasses trash).

Administrators cannot delete other users' uploads through this editor's media panel. To remove media uploaded by other users, use the standard wp-admin Media Library, where normal administrator capabilities apply.

17 Installation & Setup

  1. Upload and activate ProjectLens first. On activation, the pl_project CPT and taxonomies are registered, rewrite rules are flushed, and default role capabilities are stamped (Section 8).
  2. Upload and activate ProjectLens Project Editor. If ProjectLens is not active, you'll see an admin notice and the add-on will not initialise.
  3. Go to Projects → Global Settings and configure: default styling, archive listing, map tile provider, navigation menus, and uninstall data-retention preference.
  4. (Optional) Click Install Demo Project to preview the layout with sample content.
  5. Create your first real project under Projects → Add New Project (Section 18).
  6. (Optional) Create a frontend page containing the [pl_project_editor] shortcode and add it to your site navigation for non-admin editors.
  7. Go to Projects → Manage Access to grant pl_edit_projects to staff who will use the frontend editor (Section 19).

18 Day-to-Day Admin Workflow

Creating a New Project

  1. Go to Projects → Add New Project.
  2. Enter a title and set a featured image (used as the header background and archive card thumbnail).
  3. Fill in Project Settings & Funding Information — at minimum, Project Status, Short Description, and Location/Dates.
  4. Add segments in Project Segments: choose a type, fill in the fields, drag to reorder, uncheck "Visible" to hide a segment without deleting it.
  5. (Optional) Adjust Styling & Appearance for this project, or leave blank to inherit Global Settings defaults.
  6. Assign pl_category, pl_status, and pl_tag terms as needed for archive filtering.
  7. Publish.

19 Granting Editor Access

To let a staff member use the [pl_project_editor] page on the frontend:

  1. Ensure the user has a WordPress account and can log in.
  2. Go to Projects → Manage Access → Users.
  3. Find the user and check Edit (pl_edit_projects). This is the minimum requirement to use the frontend editor at all.
  4. Optionally check Create or Delete if they also need wp-admin abilities (these do not unlock additional frontend-editor abilities).
  5. Save access settings.
  6. Share the URL of the page containing the [pl_project_editor] shortcode with the user.

Tip: to grant access to an entire role at once (e.g. all "Contributors"), use the Role defaults tab instead of editing users individually.

What a Non-Admin Editor Will See

  • A dropdown of all non-locked projects (published, draft, or private).
  • A read-only title field.
  • The "Project Segments" tab with full editing of existing segment content, but no add/remove/reorder controls.
  • No "Project Abstracts" tab.
  • A "Project Settings" tab with only the feature image control.
  • Document segments where the download/viewer toggle checkboxes are hidden (their saved state is preserved automatically).

20 Locking Projects

Each project has a Lock from Frontend Editing checkbox in the Styling & Appearance meta box in wp-admin, stored as post meta _pl_locked.

Effect of Locking a Project

  • The project disappears entirely from the frontend editor's project dropdown for everyone except administrators.
  • Administrators still see it in the dropdown, marked with a 🔒 icon, and can still save changes to it from the frontend editor.
  • If a non-administrator somehow attempts to save a locked project, the ple_save_project AJAX call returns a 403 error: "This project is locked and cannot be edited from the frontend."
  • wp-admin editing is not affected by this flag — administrators and editors with pl_edit_projects can always edit a project in wp-admin regardless of its lock status.

When to Use Locking

Lock a project once its content is finalised and you want to prevent accidental edits from frontend collaborators, while still allowing administrators to make corrections through either wp-admin or the frontend editor.

22 FAQ

Can I use my theme's header and footer on project pages? ›

No — both the single project template and the archive template are full, standalone HTML documents that call wp_head()/wp_footer() but do not load the theme's header.php/footer.php. Use the optional ProjectLens nav menus (Section 10) for top-level navigation instead.

How do I create a new project from the frontend? ›

You can't — the [pl_project_editor] shortcode intentionally does not support creating or deleting projects, for any role. New projects must be created in wp-admin under Projects → Add New Project by a user with pl_create_projects.

What's the difference between the project_status field and the pl_status taxonomy? ›

project_status (in Project Settings) is a single value per project used for the colored status badge on the header and archive cards. pl_status is a separate, independent taxonomy used purely for the archive's status filter pills. They are not automatically synchronised — assign both if you want consistent filtering and badging.

Can non-administrators change a project's colors or fonts? ›

No. The Styling & Appearance meta box is only available in wp-admin and is not exposed by the frontend editor for any role.

Why can't an administrator delete a colleague's uploaded photo from the frontend editor? ›

The ple_delete_media AJAX handler enforces post_author === current_user_id for every request with no role-based override. This is a deliberate scoping decision to prevent any user from accidentally removing media another user or project may still depend on. Use wp-admin → Media for cross-user deletions.

How are word limits applied to descriptive segments? ›

The summary view of a descriptive segment strips all HTML from content and truncates it to word_limit words (project-level setting, falling back to the Global Settings default of 60), appending "…" if truncated. The full detail_content is shown in full in the detail panel — it is never truncated.

What happens if I deactivate ProjectLens Project Editor but keep ProjectLens active? ›

The [pl_project_editor] shortcode stops working and its AJAX endpoints become unavailable. All project data remains intact and fully editable via wp-admin, since it is stored entirely by the core ProjectLens plugin.

What happens if I deactivate ProjectLens but keep ProjectLens Project Editor active? ›

On the next page load, ple_check_dependency() detects that PL_VERSION is undefined, shows the "requires ProjectLens" admin notice, and the editor plugin does not initialise — its shortcode and AJAX handlers are not registered.

Will uninstalling the plugins delete my project content? ›

Only if Projects → Global Settings → Data & Uninstall → "Delete all data on uninstall" is checked at the time ProjectLens is deleted (not merely deactivated). By default this is unchecked, so projects, taxonomy terms, and settings are preserved if you reinstall later.

Can I have different navigation menus on different project pages? ›

No — the pl_single_nav and pl_archive_nav menu locations are each site-wide, assigned once in Global Settings. All single project pages share the same single-page menu, and the archive uses its own separate menu.
