LexiPic

Heritage Language Archive — Technical Documentation

Record, archive, and share heritage language word sets with photo, audio, native script, and Roman transliteration.

LexiPic v1.2.2 Requires WordPress (REST API) Requires PHP 8.0+ FFmpeg optional

01 Plugin Overview

LexiPic is a WordPress plugin for capturing and archiving heritage-language vocabulary: each entry pairs a photo with a spoken recording, a native-script word, and a Roman transliteration. Entries are grouped into sets (e.g. "Domestic Animals", "Kitchen Items") and rendered on the front end through a single shortcode.

The plugin was adapted from a standalone IndexedDB-based progressive web app — the REST API and import/export formats still carry traces of that lineage (see Troubleshooting and the legacy key-mapping in Lexipic_File_Handler::import_json()).

Three building blocks make up the system:

  • A custom database layer — two tables (sets, entries) storing images as binary BLOBs directly in MySQL rather than as Media Library attachments.
  • A REST API under the lexipic/v1 namespace that the front-end JavaScript app talks to exclusively — there is no page-reload form submission anywhere in the plugin.
  • A bundled third-party IME engine ("CompLing AI") that transliterates between Roman text and native script live as the user types, driven by four JSON data files per language.

Third-party engine. The transliteration engine itself (public/js/ime/engine.js, exposing the global CPLAI_ENGINE) ships as a black-box asset. LexiPic's job is to supply it with the right JSON maps and settings per language — it does not modify the engine's internals.

LexiPic creates two custom tables on activation (Lexipic_DB::create_tables(), run via dbDelta()) rather than using WordPress posts or the Media Library.

Images live in the database, not the Media Library. Lexipic_DB::get_entries() deliberately excludes the image_data column to keep list queries fast; the blob is only fetched by get_entry() when an image is actually being served or exported. Large archives with many high-resolution images will grow your database, not your uploads folder.

04 REST API Reference

All routes live under the namespace lexipic/v1, registered in Lexipic_REST_API::register_routes(). The front-end script is the only consumer in practice, but routes are usable by any authenticated client.

MethodRoutePurpose
GET/setsList all sets with an attached entry_count per set.
POST/setsCreate a set. Body: name (required), language (default bpm).
GET/sets/{id}Get one set's metadata.
DELETE/sets/{id}Delete a set, its entries, and their audio files. Owner or admin only.
GET/sets/{set_id}/entriesList entries for a set (image/audio served via separate URLs, blob excluded).
POST/sets/{set_id}/entriesCreate an entry. Body: word_script, word_roman, description, image (data-URI), audio (data-URI).
DELETE/entries/{id}Delete one entry and its audio file.
GET/entries/{id}/imageStream the stored image blob with its MIME type.
GET/entries/{id}/audioStream the audio file with byte-range support (seekable <audio> playback).
GET/sets/{id}/exportDownload a self-contained JSON export (base64 image + audio embedded). Gzipped when the client accepts it.
POST/importImport a JSON export (or legacy PWA array) as a new set.

Why Audio Has Its Own Streaming Endpoint

Audio is served through a stable per-entry URL rather than a direct link to the on-disk filename. This matters because the background FFmpeg job (see Section 7) renames the file from .webm to .mp3 and deletes the original once conversion succeeds — a player that already loaded the REST URL keeps working without the entry list needing to be refreshed.

05 Permissions & Access Control

LexiPic does not introduce custom WordPress capabilities. Access is governed by login state plus a single plugin setting, checked at two separate layers.

REST Layer

CallbackLogic
can_readTrue if lexipic_settings['public_view'] is enabled, otherwise requires is_user_logged_in().
can_writeRequires is_user_logged_in() — any logged-in user can create entries and sets via the API.
Set deletionAdditionally requires current_user_can('manage_options') or that the requester's user ID matches the set's created_by.

Frontend Tab Layer

Independently of the REST permissions above, public/partials/app.php decides which tabs to even render, based on lexipic_settings['tab_access'] — a setting holding two arrays of WordPress user IDs, one per restricted tab:

  • Archive — always shown to anyone who passes can_view_archive (logged in, or public view is on).
  • Add Entry — shown only to administrators or users whose ID appears in tab_access.add_entry.
  • Sets — shown only to administrators or users whose ID appears in tab_access.sets.

The tab bar itself ($show_tab_nav) only renders at all once more than one tab is visible — a user with no special access simply sees the Archive panel with no tab buttons.

UI restriction, not a security boundary. Hiding the Add Entry / Sets tabs only controls what renders in app.php. The REST can_write callback grants entry/set creation to any logged-in user regardless of tab access — a user without the Add Entry tab visible could still POST to /sets/{id}/entries directly if they construct the request themselves. Tab access is a convenience UI gate, not an authorization layer.

06 Image Capture & Storage

Images never touch the server as raw file uploads — they are downscaled in the browser and submitted as base64 data-URIs inside the entry-creation JSON body.

  1. The user selects a file via #lp-image-input; lexipic-app.js reads it with FileReader.
  2. An offscreen <canvas> redraws the image scaled down so neither dimension exceeds MAX_IMAGE_DIM (400px), preserving aspect ratio and never scaling up.
  3. The canvas is exported as a JPEG data-URI at quality 0.7 via canvas.toDataURL('image/jpeg', 0.7) and shown in the preview.
  4. On save, that data-URI is sent as the image field; Lexipic_File_Handler::decode_image() strips the data:<mime>;base64, header and stores the raw binary in image_data.
  5. The image is served back through GET /entries/{id}/image, which streams the blob with the stored MIME type and a 24-hour Cache-Control header.

07 Audio Capture & Conversion

Audio capture uses the browser's MediaRecorder API (always audio/webm) running in parallel with the Web Speech API for transcription — both are started together when the user presses Speak.

Capture-Time Flow

SpeechRecognition drives the live text result (locale from lexipicConfig.speechLocale), while MediaRecorder independently captures the same microphone stream as a .webm blob. When recognition.onresult fires, the transcript fills the script-word input and the recorder is stopped — the recognized text and the recorded clip are two separate artifacts from the same utterance, not derived from each other.

Server-Side Conversion (FFmpeg)

Lexipic_Audio_Converter detects FFmpeg by running ffmpeg -version and caching the result in a one-hour transient. If available, every newly saved or imported entry's audio is scheduled for background conversion via wp_schedule_single_event(), firing 5 seconds later on the lexipic_convert_audio cron hook.

StepDetail
Commandffmpeg -i {webm} -vn -ar 44100 -ac 1 -b:a 64k {mp3} — mono, 44.1kHz, 64kbps.
On successLexipic_DB::update_audio() points the entry at the new .mp3 path with MIME audio/mpeg; the original .webm is deleted.
On failureSilently leaves the original .webm in place — playback still works, just without the format normalization.

WP-Cron caveat. The conversion job relies on WordPress's pseudo-cron, which only fires on a site visit after the scheduled time. Low-traffic sites without a real system cron configured may see a short delay before the .webm → .mp3 swap happens, even though the file is already usable in the meantime.

08 IME Engine Overview

LexiPic bundles a third-party transliteration engine, internally referred to as "CompLing AI", exposed on the page as the global window.CPLAI_ENGINE with a single entry point: bpmTransliterate(text, direction, maps, preHooks, loopHooks, postHooks, opts, settings).

The engine itself ships unmodified in public/js/ime/engine.js. What LexiPic actually manages is the data fed into it — four JSON files per language, resolvable per language code, with admin tooling to replace, validate, back up, and restore them.

Resolution Order

For any language code and file key, Lexipic_IME_Manager::active_file_path() checks a saved override first:

wp-content/uploads/lexipic/ime/{lang_code}/{filename}.json   ← checked first
public/js/ime/{filename}.json                                 ← fallback (bundled BPM defaults)

This means every newly added language works immediately using the bundled BPM engine data, even before an administrator uploads anything custom for it — only languages with an explicit override saved diverge from that shared baseline.

09 The Four Engine Files

KeyFilenamePurpose
forwardforwardMap.jsonRoman → Script character/sequence mapping, used while typing in the Roman input.
reversereverseMap.jsonScript → Roman mapping, used while typing in the script input (and after speech recognition fills it).
hooksdefaults.jsonArray of "snippet hook" objects — small JS functions injected into the engine's pre/loop/post processing stages.
settingsjs-engine-settings.default.jsonEngine-wide behaviour: virama character, ZWJ/ZWNJ handling, inherent vowel, section processing order.

Map files (forward / reverse) are expected to contain six sections, checked non-strictly on upload:

  • independent_vowels, consonants, matras, special, digits, punctuation

Settings files are expected to define virama, zwj, zwnj, inherent_vowel, and section_order.

Missing sections don't break anything. Lexipic_IME_Manager::validate_structure() only produces non-blocking warnings — the engine tolerates absent sections by skipping them, and the upload still saves. Use the warnings as a guide for what transliteration behaviour will simply not work, not as a hard validation gate.

10 Language & File Management

The IME Engine admin page (admin/ime-manager-page.php, under LexiPic → IME Engine) is the control surface for everything in Lexipic_IME_Manager.

Language Registry

Languages are stored in the lexipic_languages option, seeded on first access with three built-ins:

CodeLabelSpeech Locale
bpmBishnupriya Manipuri (ইমার ঠার)bn-IN
asAssamese (অসমীয়া)as-IN
bnBengali (বাংলা)bn-IN

New languages require a code (2–20 lowercase letters/digits/underscores, starting with a letter), a display label, an optional BCP-47 speech locale (defaults to bn-IN), and an optional source language to clone the four active files from. Built-in languages cannot be deleted, though their engine files can still be replaced.

Replacing a File

Uploading a new JSON file for a language/key (Lexipic_IME_Manager::replace_file()) runs through several checks before saving:

  1. 5MB size limit; .json extension required if one is present on the uploaded filename.
  2. The file must parse as valid JSON — invalid JSON is rejected outright with the decode error.
  3. Structural validation (Section 9) runs and produces warnings, but does not block the save.
  4. Whatever was previously active (override or bundled fallback) is snapshotted to the backups folder.
  5. The uploaded JSON is re-encoded with JSON_UNESCAPED_UNICODE | JSON_PRETTY_PRINT — not written byte-for-byte — which normalizes formatting and strips stray BOM/control characters.

Backups

Every replace, revert, or restore action calls backup_current() first, snapshotting to uploads/lexipic/ime/{lang}/backups/{key}_{timestamp}.json. Backups are capped at 20 per file key per language (MAX_BACKUPS_PER_FILE); the oldest are pruned automatically once the cap is exceeded. The admin page lists the 15 most recent backups across all four keys, each restorable with one click — restoring itself snapshots the current state first, so a restore can always be undone.

Downloads

Each file can be downloaded individually, or all four active files for a language can be bundled into a single zip via stream_zip_download() — this requires the PHP ZipArchive extension; if it's unavailable, the plugin reports this clearly rather than failing silently, and individual downloads remain unaffected.

11 Snippet Hooks

The hooks file (defaults.json) is a JSON array of objects, each describing one JavaScript function the engine should run at a specific processing stage:

FieldMeaning
snippet_keyIdentifier for the hook (for display/debugging).
hook_stageOne of pre, loop, or post — when in the transliteration pass the function runs.
directionforward or reverse — which transliteration direction the hook applies to.
js_bodyA full named JavaScript function declaration as a string, e.g. function myHook(...) {...}.
is_activeHooks with a falsy value here are skipped entirely.
sort_orderDetermines execution order within the same stage.

On page load, lexipic-app.js partitions imeHooks by direction and stage (getHooksForDirection()), then compiles each js_body string into a callable function using the Function constructor, extracting the function name by regex to invoke it. Compilation failures for an individual hook are caught and logged to the console without affecting the others.

Hooks execute arbitrary JavaScript. Because js_body is compiled with new Function() and run directly in the visitor's browser, anyone with edit access to a language's hooks file (administrators, via the IME Engine page) can run arbitrary client-side JavaScript on every page where the shortcode appears. Treat the hooks file the same way you would treat direct theme/plugin-file edit access.

12 Shortcode & Tabs

The single shortcode [lexipic] renders the entire app. It accepts one optional attribute:

AttributeDefaultEffect
set(empty)A set slug to pre-select in the set dropdown and load on page load, e.g. [lexipic set="domestic-animals"].

The shortcode callback enqueues style.css, the IME engine script, and lexipic-app.js (in that dependency order — the engine must load before the app script that calls it), then localizes a lexipicConfig object onto the page containing the REST URL, nonce, speech locale, active language, FFmpeg availability flag, the write-permission flag, and the resolved IME maps/hooks/settings for the active language.

Tab Visibility

Three tabs exist — Archive, Add Entry, Sets — but which ones render depends on the access rules in Section 5. The tab bar itself only appears once more than one tab qualifies; otherwise the page is just the Archive panel with no chrome around it.

13 Frontend Entry Flow

End-to-end, adding an entry from the Add Entry tab follows this sequence:

  1. User selects/takes a photo → downscaled and previewed as a JPEG data-URI (Section 6).
  2. User presses Speak → SpeechRecognition and MediaRecorder start together against the same microphone stream.
  3. On recognized speech, the transcript fills the script-word field and fires an input event, triggering reverse transliteration (script → Roman) to auto-fill the Roman field via the IME engine.
  4. The user may instead type directly into either field — typing in the Roman field runs forward transliteration live and mirrors the script-equivalent into the script field; typing in the script field runs reverse transliteration into the Roman field.
  5. On save, the recorded audio blob is converted to a base64 data-URI and POSTed alongside the image data-URI and both text fields to POST /sets/{set_id}/entries.
  6. The archive card list (#lp-archive-slider) refreshes immediately from GET /sets/{set_id}/entries, and the audio-conversion cron job is scheduled in the background if FFmpeg is available.

Usernames, not display names. The tab-access fields take WordPress login names. lexipic_sanitize_settings() looks up each comma-separated token with get_user_by('login', …), falling back to a numeric-ID lookup for values saved by older versions. Any token that resolves to no user produces an admin notice listing the unmatched name(s), and that entry is dropped rather than silently saved as garbage.

15 Import / Export

Found under LexiPic → Import / Export (admin/import-export.php), this page provides a simple form-based alternative to the REST /import and /sets/{id}/export endpoints — useful for administrators who prefer a GUI over constructing API calls.

Export Format

Each export is a single self-contained JSON file — image and audio are embedded as base64 data-URIs, so the file has no external dependencies and can be archived or emailed as-is:

{
  "lexipic_version": "1.2.2",
  "set": { "name": "...", "slug": "...", "language": "bpm" },
  "entries": [
    {
      "id": 1,
      "word_script": "...",
      "word_roman": "...",
      "description": "...",
      "image": "data:image/jpeg;base64,...",
      "audio": "data:audio/mpeg;base64,...",
      "image_mime": "image/jpeg",
      "audio_mime": "audio/mpeg"
    }
  ]
}

Legacy Format Support

Because LexiPic descends from a standalone PWA, import_json() also accepts the old flat-array format (no top-level set key) and maps its legacy field names (heritage, transliteration) onto the current schema automatically — older exports continue to import without modification.

Imports create a new set. Entry IDs from the source file are never reused — every import inserts a fresh set and fresh entry rows. Importing the same file twice will produce two duplicate sets, not an update to the first.

16 Installation & Setup

  1. Upload and activate the plugin. Activation runs Lexipic_DB::create_tables(), creates wp-content/uploads/lexipic/ (with an .htaccess blocking directory listing) and its ime/ subfolder, and flushes rewrite rules so REST routes resolve.
  2. Visit LexiPic → Settings to confirm the upload directory is writable and check the FFmpeg status badge.
  3. Add the [lexipic] shortcode to any page or post.
  4. As an administrator, open the page, switch to the Sets tab (visible to admins by default), and create your first set, choosing a language.
  5. Grant other contributors access to Add Entry and/or Sets via their usernames in LexiPic → Settings if you want collaborators who aren't administrators.

Server Requirements

  • PHP 8.0+, with the exec() function available if you want server-side MP3 conversion (some managed hosts disable it).
  • FFmpeg installed and on the system PATH — optional; without it, audio stays as .webm, which Safari/iOS may not play.
  • The ZipArchive PHP extension — optional, only needed for the "download all four files as zip" convenience in the IME Engine page.
  • HTTPS — both getUserMedia() (microphone) and SpeechRecognition require a secure context in modern browsers; the plugin's microphone-access error message reminds the user of this directly.

17 Uninstall Behaviour

Deleting the plugin from the Plugins screen (not merely deactivating it) runs uninstall.php, which is destructive and irreversible:

  • Drops the wp_lexipic_entries and wp_lexipic_sets tables entirely.
  • Deletes the lexipic_settings and lexipic_languages options, plus the FFmpeg-detection transient.
  • Recursively removes the entire wp-content/uploads/lexipic/ directory — every audio file, every custom IME override, every backup.
  • Clears any pending lexipic_convert_audio scheduled cron events.

There is no data-retention toggle. Unlike plugins that offer a "delete data on uninstall" checkbox, LexiPic's uninstall.php unconditionally removes everything the moment the plugin itself is deleted. Export anything you want to keep (Section 15) before deleting the plugin — deactivating it alone leaves all data and files untouched.

19 FAQ

Are images stored in the WordPress Media Library? ›

No. Images are stored as binary MEDIUMBLOB data directly in the wp_lexipic_entries table and served through a dedicated REST endpoint — they never become Media Library attachments and won't appear in Media → Library.

Can a non-admin delete a set they didn't create? ›

No. delete_set in the REST API explicitly checks that the requester is either an administrator (manage_options) or the set's original creator. Anyone else gets a 403.

What happens if I add a language but never upload custom engine files for it? ›

It works immediately using the bundled BPM defaults — active_file_path() falls back to public/js/ime/*.json for any language with no saved override. Transliteration simply behaves identically to BPM until you upload language-specific files.

Is the tab access control a real security restriction? ›

No — it only controls which tabs render in the shortcode's HTML. The REST can_write permission callback allows any logged-in user to create entries and sets regardless of tab visibility. Treat tab access as a UI convenience, not an authorization mechanism.

Can I edit the snippet hooks safely? ›

Hooks are compiled with JavaScript's Function constructor and executed directly in every visitor's browser, so they amount to arbitrary client-side code execution. Only grant IME Engine access (which requires manage_options) to administrators you'd trust with direct file-edit access to the site.

What happens to my data if I deactivate the plugin? ›

Nothing is removed. Deactivation only unhooks the plugin's actions and filters; the database tables, options, and uploaded files (audio, custom IME overrides, backups) are left exactly as they were. Data is only deleted if you delete the plugin entirely (see Section 17).

Why are there two different transliteration directions? ›

Because entries can originate from either input method. Speech recognition produces script text first, which needs reverse transliteration (script → Roman) to populate the Roman field; a contributor typing in Roman directly needs forward transliteration (Roman → script) to populate the script field as they type.

Can I import a JSON file from the original standalone PWA version? ›

Yes. Lexipic_File_Handler::import_json() detects the absence of a top-level set key and treats the payload as a legacy flat array, remapping the old heritage and transliteration field names to word_script and word_roman automatically.
