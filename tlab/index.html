TransLitLab

Technical Documentation & Administrator Guide

Browser-based test and configuration lab for the CompLing-AI transliteration engine — dual JS/PHP testing, lexicon editing, snippet rules, and per-user access control.

TransLit-Lab v1.1.0 Requires WordPress 6.0+ Requires PHP 8.0+ Tested up to WP 6.7 License: GPLv2+

1 Plugin Overview

TransLit-Lab is a self-contained WordPress plugin that provides a browser-based lab for testing and configuring the CompLing-AI transliteration engine. It is primarily aimed at linguists, developers, and administrators working on Bengali/BPM (Bengali Phonetic Mapping) transliteration.

The lab is delivered entirely through a single frontend shortcode ([translitlab]) and communicates with the server exclusively via WordPress AJAX actions — there are no standalone API endpoints or direct file access required.

Key Capabilities

  • Dual-engine testing — run the same input through the JavaScript (browser) and PHP (server) engines side by side and compare output and step-by-step processing logs.
  • Snippets editor — create, reorder, enable/disable, import, and export pre/loop/post processing rules (regex or literal) per direction.
  • Lexicon map manager — view, search, add, edit, delete, and reorder forward (Roman → BPM) and reverse (BPM → Roman) transliteration map entries, organized into named sections.
  • Engine settings — configure virama, ZWJ/ZWNJ, inherent vowel, non-linking consonants, and section ordering per engine independently.
  • Debug panel — inspect the step-by-step processing log from the last run for each engine.
  • Reset to defaults — restore maps, snippets, or engine settings to the plugin's bundled defaults at any time.
  • Role-based access control — a dedicated "Transliterator" role, admin-managed approval list, and per-user (or global) tab visibility settings.

4 Transliteration Engine

TransLit-Lab ships twin engines — a PHP class and a JavaScript module — designed to produce bit-identical output from the same inputs. The Test tab is the primary surface for cross-validating the two.

PHP Engine (TranslitEngine)

Defined in includes/class-engine.php. Has no WordPress dependency and can be instantiated standalone.

$engine = new TranslitEngine( $forward_map, $reverse_map, $settings );
$result = $engine->transliterate( $input, $direction, $snippets );
// Returns: [ 'output' => string, 'log' => string[] ]
ParameterTypeDescription
$forward_maparrayAssociative array of sections → entry pairs (Roman → BPM).
$reverse_maparrayAssociative array of sections → entry pairs (BPM → Roman).
$settingsarrayEngine configuration (see Engine Settings section).
$inputstringRaw text to transliterate.
$directionstring'forward' (Roman → BPM) or 'reverse' (BPM → Roman).
$snippetsarrayArray of snippet objects applied as pre/loop/post hooks.

JavaScript Engine (engine.js)

The JS engine runs entirely in the browser with no server round-trip. It is enqueued as translitlab-engine and consumed by app.js. The JS engine is executed on the same map/snippet/settings data loaded from the server at page init.

Both engines process snippets in the same pipeline: pre hooks → map substitution loop → loop hooks (per character/token) → post hooks. Differences in output between the two engines indicate either a parity bug or differing settings/map data.

5 Lexicon Maps

The plugin maintains two maps — forward (Roman → BPM) and reverse (BPM → Roman) — stored in separate database tables. Each map is organized into named sections that also control substitution priority.

Map Sections

The default section order used by both engines is: special → consonants → matras → independent_vowels → digits → punctuation. This order can be overridden per engine in Engine Settings.

Map Entry Structure

Each entry is a tuple of [key, value] where both are strings. Entries within a section are ordered by sort_order (ascending), which directly controls substitution priority — earlier entries are tried first.

Reordering Entries

The Maps tab supports drag-and-drop reordering within a section. On save, the plugin performs a two-step update to avoid UNIQUE constraint collisions: it first shifts all sort_order values up by 1,000,000, then writes the new order. This ensures no transient duplicates during the update.

Saving a map replaces the entire table for that direction (DELETE then re-insert). Export your map as JSON from the Maps tab before making bulk edits.

6 Snippets (Processing Rules)

Snippets are named processing rules injected into the engine pipeline at a specific direction and hook stage. Both JS and PHP bodies of a snippet are stored together, so each snippet runs identically on both engines.

Snippet Fields

FieldTypeDescription
snippet_keyVARCHAR(100)Unique identifier for the snippet. Used as the database primary key.
labelVARCHAR(255)Human-readable name shown in the UI.
directionVARCHAR(20)forward or reverse.
hook_stageVARCHAR(20)pre, loop, or post. Controls when the snippet runs in the pipeline.
sort_orderINTExecution order within the same direction + stage group.
is_activeTINYINT(1)1 = active (runs during transliteration), 0 = disabled.
sourceVARCHAR(100)'custom' for user-created, or another identifier for bundled snippets.
logic_descriptionTEXTPlain-language explanation of what the snippet does. Not executed.
js_bodyLONGTEXTJavaScript function body executed by the JS engine.
php_bodyLONGTEXTPHP function body executed by the PHP engine.

Saving Snippets

On save, the plugin performs an upsert: existing snippets with matching snippet_key are updated, snippets absent from the incoming payload are deleted. A sort_order bump (+1,000,000) prevents UNIQUE collisions during the reorder phase, identical to the map reorder strategy.

7 Engine Settings

Each engine (JS and PHP) has its own independent set of settings, stored in the wp_translit_settings table keyed by engine ('js' or 'php').

Setting KeyDefaultDescription
viramaU+09CDUnicode virama (halanta) character appended to consonants to suppress the inherent vowel.
zwjU+200DZero-width joiner — used to force ligature formation between characters.
zwnjU+200CZero-width non-joiner — used to explicitly break ligature formation.
inherent_vowel'o'The Roman character representing the inherent vowel of a consonant (e.g. "k" → "ko").
non_linking_consonants["r","R","ng","M","NG","Ng",":","H","^","~"]Consonants that never form clusters with a following consonant.
hardcode_independent_voweltrueWhen true, independent vowel forms are emitted at word-start and after non-linking consonants.
hardcode_explicit_zwnjtrueWhen true, an explicit ZWNJ in the input forces the engine to break a cluster.
hardcode_consonant_matratrueWhen true, the engine applies consonant-matra rules for vowel diacritics.
hardcode_consonant_clustertrueWhen true, the engine applies consonant cluster (hasanta) rules.
section_order["special","consonants","matras","independent_vowels","digits","punctuation"]Order in which map sections are applied during substitution. Earlier sections take priority.

8 Roles & Access Control

Access to the lab follows a two-gate model: a user must first hold the Transliterator role, and must additionally be on the approved-users list. Administrators bypass both gates.

Access Decision (TranslitLab_Role::user_can_access())

  • User is not logged in → lab shows a login prompt.
  • User has manage_options capability (administrator) → always has access.
  • User holds the Transliterator role AND is in translitlab_approved_users option → has access.
  • Any other case → lab shows a permission notice.

Transliterator Role

The role (slug: transliterator, display: "Transliterator") is registered on plugin activation via register_activation_hook → TranslitLab_Role::create_role(). It grants only the read capability so users can log into WordPress. The role is removed permanently on plugin deletion (uninstall.php).

On deactivation (not deletion), the role is intentionally not removed — user accounts remain intact so reactivating the plugin restores access immediately.

Granting Access

Go to Settings → TranslitLab. For each non-admin user, check Transliterator Role to assign the role, then check Lab Access to add them to the approved list. Both checkboxes must be active for access to be granted. The Lab Access checkbox is disabled unless the user has the role.

9 Tab Visibility Control

Administrators can control which of the five tabs are shown in the lab, both globally and per individual user.

Global Default Tabs

Stored as the translitlab_default_tabs WordPress option — an array of tab slugs (e.g. ["test","snippets","maps"]). Configured from the "Default Tab Visibility" card in Settings → TranslitLab. Affects all Transliterator users who do not have a per-user override.

Per-User Override

Each user can have a custom tab list stored in user meta under the key translitlab_tabs. Enable the "Override defaults" checkbox for a user in the settings table and select their allowed tabs. If the checkbox is unchecked, any previously saved user meta is deleted and the global default applies.

Resolution Order (TranslitLab_Role::user_tab_config())

  • Per-user meta (translitlab_tabs) is checked first.
  • Falls back to the translitlab_default_tabs option.
  • Falls back to all five tabs enabled if neither is set.

The resolved tab list is passed to the frontend as TranslitLabConfig.enabledTabs via wp_localize_script, and the lab JS uses this to skip event binding on hidden tabs.

11 AJAX Endpoints

All frontend communication uses WordPress AJAX (admin-ajax.php). Every action requires a logged-in, authorized user and a valid translitlab_nonce nonce. There are no nopriv variants — all actions require login.

The JS sends a JSON payload as a payload FormData field (URL-encoded JSON string). The PHP AJAX handler decodes this field; a raw JSON body fallback also exists for direct API use.

Action NameMethodDescription
translitlab_loadGET-style POSTReturns all snippets, forward map, reverse map, PHP settings, and JS settings in one payload. Called on lab init.
translitlab_run_phpPOSTRuns the PHP engine with the provided input, direction, snippets, maps, and settings. Returns output and log.
translitlab_get_engine_sourceGET-style POSTReturns the raw source of engine.js and class-engine.php as strings (for the Debug tab source viewer).
translitlab_save_snippetsPOSTReplaces the entire snippets table with the submitted array (upsert + delete orphans).
translitlab_save_mapPOSTReplaces the entire specified map (forward or reverse) with the submitted data.
translitlab_save_engine_settingsPOSTUpserts settings for the specified engine (js or php).
translitlab_reset_defaultsPOSTResets the specified component (forward, reverse, snippets, js_settings, php_settings, or all) to bundled defaults. Returns updated data when resetting settings.
translitlab_reorder_map_sectionPOSTReorders entries within a single section of a map. Accepts type, section, and ids (ordered array of row IDs).
translitlab_check_sort_order_conflictsPOSTChecks for sort order conflicts in a map (currently returns empty conflicts — reserved for future use).

Auth Guard

Every AJAX handler calls TranslitLab_Ajax::guard() first, which verifies is_user_logged_in(), TranslitLab_Role::user_can_access(), and check_ajax_referer('translitlab_nonce', 'nonce'). A failure returns a 403 JSON error response and halts execution.

12 Default Data & Reset

All default data is bundled as JSON files in the data/ directory and seeded into the database on plugin activation — but only if the snippets table is empty. This prevents activation from overwriting existing data.

Activation Seeding

On activation, TranslitLab_DB::install() creates the tables then calls maybe_seed(). If wp_translit_snippets has zero rows, all five data files are imported.

Reset Options

The translitlab_reset_defaults AJAX action accepts a which parameter:

which valueWhat is reset
forwardForward map only (from data/forwardMap.json).
reverseReverse map only (from data/reverseMap.json).
snippetsAll snippets (from data/defaults.json).
js_settingsJS engine settings (from data/js-engine-settings.default.json).
php_settingsPHP engine settings (from data/php-engine-settings.default.json).
allEverything above.

Reset is destructive and immediate — existing customizations are replaced. Export your maps and snippets as JSON before using reset.

13 Installation & Setup

Installation

  • Upload the translitlab folder to /wp-content/plugins/, or install the plugin ZIP through Plugins → Add New → Upload Plugin.
  • Activate the plugin through the Plugins menu in WordPress.

What Activation Does

  • Registers the Transliterator custom role (if it doesn't already exist).
  • Creates four database tables via dbDelta().
  • Seeds the tables with bundled defaults if they are empty.

Initial Configuration

  • Go to Settings → TranslitLab to grant users the Transliterator role and lab access.
  • Optionally configure the global default tab visibility.
  • Add the [translitlab] shortcode to any page or post to display the lab to authorized users.

Create a dedicated private page (e.g. "Transliteration Lab") and restrict it to logged-in users via your theme or a membership plugin for the cleanest access experience.

14 Shortcode Reference

The plugin registers a single shortcode:

[translitlab]

The shortcode accepts no attributes. Behavior depends on the current user's state:

User StateOutput
Not logged inA <p class="translitlab-notice"> message: "Please log in to access the Transliteration Lab."
Logged in, no accessA <p class="translitlab-notice"> message: "You do not have permission to access the Transliteration Lab."
Logged in, has accessThe full lab UI HTML with all permitted tabs.

Assets (style.css, engine.js, app.js, JSZip CDN) are conditionally enqueued only on pages that contain the shortcode, detected via has_shortcode() on wp_enqueue_scripts. The lab config (TranslitLabConfig) is injected via wp_localize_script with the AJAX URL, nonce, and enabled tabs.

15 Uninstall Behaviour

Deactivating the plugin leaves all data intact — the shortcode stops working, but tables, options, user meta, and the role are preserved. Reactivating restores full functionality immediately.

On Deletion

When the plugin is deleted (not just deactivated) through wp-admin, uninstall.php runs and performs the following actions permanently:

  • Drops all four plugin tables: wp_translit_forward_map, wp_translit_reverse_map, wp_translit_snippets, wp_translit_settings.
  • Deletes the translitlab_approved_users option.
  • Deletes the translitlab_default_tabs option.
  • Removes translitlab_tabs user meta from all users.
  • Removes the transliterator custom role from WordPress.

Deletion is irreversible. Export your maps and snippets as JSON from the lab before deleting the plugin if you intend to reinstall later.

16 Troubleshooting

SymptomLikely cause / fix
Shortcode shows "Please log in…"The visitor is not logged in. The lab requires an authenticated session.
Shortcode shows "You do not have permission…"The user lacks the Transliterator role, is not on the approved-users list, or both. Check Settings → TranslitLab and ensure both the Role and Lab Access checkboxes are checked for the user.
AJAX call returns 403Nonce mismatch (page cached without the nonce?), or user lost access between page load and request. Refresh the page. If caching is the culprit, exclude the lab page from your cache plugin.
JS and PHP engines produce different outputEngine settings or map data may differ between engines. Open the Engines tab and compare the two settings panels. Use Reset → all to bring both back to defaults, then re-apply changes.
Tables not created / lab throws DB errorsThe activation hook may not have run (e.g., plugin was copied manually without activating). Deactivate and reactivate the plugin through wp-admin to trigger dbDelta().
Saving map returns no error but data doesn't updateThe save_map action replaces the entire table — confirm you're submitting the full map payload, not just a diff. Check the browser Network tab for the AJAX response.
Snippet reorder doesn't stickA UNIQUE KEY (direction, hook_stage, sort_order) conflict during the save. The two-step bump should prevent this, but if it occurs, try resetting snippets to defaults and re-applying your changes.
Lab page shows no tabsAll tabs may be disabled in the global default or per-user override. Check Settings → TranslitLab → Default Tab Visibility and confirm at least one tab is checked.
User meta tab override not taking effectConfirm the "Override defaults" checkbox is checked and saved for that user. If the checkbox is unchecked on save, any previous meta is deleted and the global default applies.
JSZip not loading (import/export broken)The lab loads JSZip from cdnjs.cloudflare.com. Check if a firewall or CSP policy is blocking the CDN. If so, self-host jszip.min.js and update the enqueue URL in class-frontend.php.

17 FAQ

Who can access the lab on the frontend? ›

Logged-out visitors see a login prompt. Logged-in users without the Transliterator role and explicit Lab Access approval see a permission notice. Users with manage_options (administrators) always have full access regardless of the approved list.

Do both engines always produce the same output? ›

They are designed to — the PHP engine is written to mirror the JS engine exactly. However, if their settings or map data have diverged (e.g. settings were saved for one engine but not the other), output will differ. The Test tab exists precisely to surface these differences. Use Reset → all to resync both to defaults.

Can I add the shortcode to multiple pages? ›

Yes. The shortcode can be placed on any number of pages or posts. Assets are conditionally enqueued only on pages where has_shortcode() returns true, so there's no performance impact on other pages.

What happens to my custom maps and snippets if I reset to defaults? ›

They are permanently replaced. The reset AJAX action calls TranslitLab_DB::reset_defaults(), which re-imports from the bundled JSON files — any customizations in the database are overwritten. Always export your maps and snippets as JSON from the Maps and Snippets tabs before resetting.

What happens when I deactivate the plugin? ›

Deactivation removes the Transliterator role from WordPress (so it no longer appears in user-role dropdowns) but leaves all table data, options, and user meta fully intact. The shortcode stops rendering the lab. Reactivating the plugin re-registers the role — users who held it will have it reassigned automatically by WordPress on their next login, and approved-user settings are preserved.

Will deleting the plugin lose my transliteration maps? ›

Yes — uninstall.php drops all four plugin tables. If you plan to reinstall, export your forward map, reverse map, and snippets as JSON from the lab before deleting. The exported JSON files can be re-imported into a fresh install.

Can I use this plugin without the CompLing-AI plugin? ›

Yes. TransLit-Lab is a standalone WordPress plugin with no dependency on any other plugin. It bundles its own PHP and JS engines and manages its own database tables independently.

How do I give a user access to only some tabs? ›

Go to Settings → TranslitLab, find the user in the table, check "Override defaults," and select only the tabs you want them to see. Save. Their per-user tab config is stored in translitlab_tabs user meta and takes precedence over the global default.

Is the map/snippet data safe from other plugins or theme code? ›

All data is stored in dedicated custom tables (not wp_options or wp_postmeta), and all write operations go through authenticated AJAX handlers with nonce validation. Other plugins cannot write to these tables unless they directly call $wpdb — which would be unusual outside of a hostile plugin scenario.
